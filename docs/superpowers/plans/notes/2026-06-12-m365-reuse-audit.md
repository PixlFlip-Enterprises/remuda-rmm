## Consent-Flow Reuse

`buildAdminConsentUrl` (`apps/api/src/services/c2cM365.ts:55-65`) takes `{clientId, state, redirectUri}` and builds:

```
https://login.microsoftonline.com/common/adminconsent
  ?client_id=<clientId>
  &redirect_uri=<redirectUri>
  &state=<state>
```

**Verdict: yes, reusable verbatim.** The function is fully parameterised — it does not hard-code C2C credentials or the C2C callback URI. Plan 4 can call it with `CLIENT_AI_ENTRA_CLIENT_ID` as `clientId` and a `/client-ai/auth/consent-callback` URI as `redirectUri`.

**What a `/client-ai` consent callback route needs to build:**
1. Generate a short-lived `state` GUID and store it (e.g. `c2c_consent_sessions` pattern from `apps/api/src/db/schema/c2c.ts:109-122`, or a Redis key) keyed by `state`.
2. Call `buildAdminConsentUrl({ clientId: CLIENT_AI_ENTRA_CLIENT_ID, state, redirectUri: '<base>/api/v1/client-ai/auth/consent-callback' })`.
3. On callback, verify `state`, extract `?tenant=` (concrete GUID from Microsoft), upsert `client_ai_tenant_mappings`.
4. `getCallbackUri` (`apps/api/src/services/c2cM365.ts:69-77`) is C2C-specific (hard-codes `/c2c/m365/callback` suffix) — **do not reuse**; build the client-AI callback URI inline.

## Token-Verification Reuse

**No existing code verifies *user* tokens from customer tenants.**

- `apps/api/src/services/c2cM365.ts` — client-credentials (`client_credentials` grant) only; acquires tokens on behalf of the platform app, not end-user tokens. No JWKS fetch, no JWT decode of incoming tokens (`apps/api/src/services/c2cM365.ts:98-158`).
- `apps/api/src/services/cfAccessJwt.ts` — the only remote-JWKS user-token verifier in the codebase; verifies Cloudflare Access JWTs (not Microsoft Entra ID tokens). Uses `jose` (`createRemoteJWKSet`, `jwtVerify`).
- **Verdict:** no reuse possible. Task 7's `clientAiEntraJwt.ts` must be a new service mirroring `cfAccessJwt.ts`'s structure (remote JWKS via `jose`) but targeting `https://login.microsoftonline.com/common/discovery/v2.0/keys` and binding `iss` to the concrete tenant GUID extracted from the `tid` claim.

## Tenant-GUID Validation Precedent

The canonical regex (`apps/api/src/services/c2cM365.ts:19-20`):

```
/^[0-9a-fA-F]{8}-[0-9a-fA-F]{4}-[0-9a-fA-F]{4}-[0-9a-fA-F]{4}-[0-9a-fA-F]{12}$/
```

The migration check (`apps/api/migrations/2026-05-31-c2c-tenant-id-guid-check.sql:24-25`) uses the PostgreSQL case-insensitive form:

```sql
tenant_id ~* '^[0-9a-f]{8}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{12}$'
```

**Task 2 CHECK constraint must match this pattern** (case-insensitive `~*`, standard UUID GUID form, provider-conditional or applied to all rows depending on column).

## Recommendation for Plan 4 Onboarding Wizard

**Pre-fill tenant GUID:**
```sql
SELECT tenant_id FROM m365_connections WHERE org_id = $1
UNION ALL
SELECT m365_tenant_id FROM delegant_m365_connections WHERE org_id = $1 LIMIT 1
LIMIT 1;
```
Use `m365_connections.tenant_id` as primary source (one per org, live-validated).
Fall back to `delegant_m365_connections.m365_tenant_id` if primary absent, but note there may be multiple rows — pick `LIMIT 1` (or let the user choose if >1).

**Consent URL construction:**
Reuse `buildAdminConsentUrl` from `apps/api/src/services/c2cM365.ts` with `CLIENT_AI_ENTRA_CLIENT_ID` and a new `/client-ai/auth/consent-callback` redirect URI.

**State persistence pattern:**
Mirror `c2c_consent_sessions` (`apps/api/src/db/schema/c2c.ts:109-122`) or use a Redis TTL key — tie the `state` to `orgId` + `userId` so the callback can re-associate.
